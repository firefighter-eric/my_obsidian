# LLM Wiki

一种用 LLM 构建个人知识库的模式。

这是一份想法文档，设计出来就是为了直接复制粘贴给你自己的 LLM Agent 使用（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达这套方法的高层理念；至于具体实现细节，你的 agent 会和你协作把它逐步搭建出来。

## 核心想法

大多数人对 LLM 和文档的使用方式，基本都类似 RAG：你上传一批文件，LLM 在提问时临时检索相关片段，再生成回答。这种方式当然能工作，但问题在于，LLM 每回答一次问题，都在从头重新发现知识。知识不会累积。你问一个需要综合五篇文档的细致问题，它每次都得重新找到相关片段，再重新拼起来。什么都没有真正沉淀下来。NotebookLM、ChatGPT 文件上传，以及大多数 RAG 系统，基本都是这种模式。

这里的思路不一样。与其在查询时才从原始文档里临时检索，不如让 LLM **逐步构建并维护一个持久化的 wiki**。它是一组结构化、互相链接的 Markdown 文件，位于你和原始资料之间。当你加入一个新来源时，LLM 不只是给它做索引，留待以后检索；它会真正去读它，提取关键信息，并把这些信息整合进已有 wiki 中：更新实体页、修订主题摘要、标注新数据与旧说法之间的冲突、加强或修正现有综合结论。知识被整理一次之后，会被持续维护，而不是每次提问时都重新推导。

关键差异就在这里：**这个 wiki 是一个持久存在、会不断复利增长的产物。** 交叉引用已经建立好了，冲突已经被标记出来，综合结论已经反映你读过的全部内容。每新增一个来源、每提出一个问题，这个 wiki 都会变得更丰富。

你自己几乎不用亲手写 wiki，或者说很少需要写。wiki 由 LLM 来写并维护。你负责找来源、探索问题、提出对的问题。LLM 负责所有脏活累活：摘要、交叉引用、归档、以及那些真正让知识库长期可用的维护工作。实际使用时，我会一边开着 LLM agent，一边开着 Obsidian。LLM 根据我们的对话直接修改文件，我则实时浏览结果，点开链接、看 graph view、读更新后的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这套方法可以用于很多不同场景。举几个例子：

- **个人知识管理**：跟踪自己的目标、健康、心理、自我提升，把日记、文章、播客笔记逐步归档，长期构建对自己的结构化理解。
- **研究**：围绕一个主题连续研究数周或数月，阅读论文、文章、报告，并逐步建立一个带有演化中主线判断的完整 wiki。
- **读书**：每读完一章就归档，逐步建立角色、主题、情节线索及其关联页面。读完整本书后，你会得到一个非常丰富的伴随型 wiki。可以类比 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki：成千上万条互相链接的页面，覆盖人物、地点、事件、语言，由社区志愿者多年维护而成。你完全可以在个人阅读过程中做出类似的东西，只不过由 LLM 来做所有交叉引用和维护。
- **团队 / 公司**：由 LLM 持续维护内部 wiki，输入材料来自 Slack 讨论、会议转录、项目文档、客户访谈。必要时也可以有人类参与审核更新。wiki 之所以能保持最新，是因为维护工作交给了 LLM，而不是团队里没人愿意做的手工整理。
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣主题深挖**：任何需要长期累积知识，并且你希望这些知识被组织起来而不是四处散落的场景，都适用。

## 架构

这里有三层：

**原始来源（Raw sources）**：你整理好的原始资料集合。可以是文章、论文、图片、数据文件。这一层是不可变的，LLM 只能读取，不能修改。它是你的事实源头。

**Wiki**：由 LLM 生成的一组 Markdown 文件。可以包括摘要页、实体页、概念页、比较页、总览页、综合页。LLM 完全拥有这一层：它创建页面、在新来源到来时更新页面、维护交叉引用，并保持整体一致。你负责阅读；LLM 负责写作。

**Schema**：一份文档（例如 Claude Code 用 `CLAUDE.md`，Codex 用 `AGENTS.md`），告诉 LLM 这个 wiki 的结构是什么、遵循什么约定、在 ingest 来源、回答问题、维护 wiki 时分别该走什么流程。这是关键的配置文件，它决定了 LLM 是一个有纪律的 wiki 维护者，而不是一个泛泛而谈的聊天机器人。随着你们逐渐摸索出适合你自己领域的方法，这份 schema 也会和 LLM 一起不断演进。

## 操作

**Ingest（接入来源）**。你把一个新的来源放进原始资料目录，然后告诉 LLM 去处理它。一个典型流程是：LLM 读取来源，和你讨论关键收获，在 wiki 中写出摘要页，更新索引，更新 wiki 里相关的实体页和概念页，并在日志中追加一条记录。一个来源可能会触及 10 到 15 个 wiki 页面。就我个人而言，我更喜欢一次只 ingest 一个来源，并保持参与感：我会阅读摘要、检查更新内容，并引导 LLM 重点强调什么。当然，你也可以一次批量 ingest 很多来源，只是监督更少。最终采用哪种工作流，取决于你的风格；你需要把它总结下来，写进 schema，方便未来会话延续。

**Query（查询）**。你围绕 wiki 提问题。LLM 搜索相关页面，阅读它们，并带引用地综合出答案。答案的形式可以很多样，取决于问题本身，例如一页 Markdown、一个对比表、一份幻灯片（Marp）、一张图表（matplotlib）、一个画布。这里最重要的一点是：**好的回答本身也可以被沉淀回 wiki，成为新页面。** 你提出过的比较、做出的分析、发现的关联，都是有价值的，不应该消失在聊天记录里。这样一来，你的探索过程就会像 ingest 来源一样，持续地在知识库中复利增长。

**Lint（体检 / 巡检）**。你可以定期让 LLM 给整个 wiki 做健康检查。重点看：页面之间有没有矛盾；旧结论有没有被更新来源推翻；有没有没有入链的孤儿页；某些重要概念是否被多次提到却还没有独立页面；有没有缺失的交叉引用；是否存在可以通过 web search 补齐的数据空白。LLM 很擅长据此提出下一步值得调查的问题，以及值得寻找的新来源。这能让 wiki 在规模变大时依然保持健康。

## 索引与日志

有两个特殊文件会帮助 LLM 和你在 wiki 规模增长后仍然保持可导航性。它们各自承担不同角色：

**index.md** 面向内容。它是整个 wiki 的目录：列出每个页面的链接、一句话摘要，以及可选的元信息，例如日期、来源数量等。通常按类别组织（实体、概念、来源等）。每次 ingest 时，LLM 都要更新它。回答问题时，LLM 先读索引，找到相关页面，再继续往下钻取。这个方法在中等规模下效果 surprisingly 好（例如大约 100 个来源、几百个页面），而且可以避免一开始就引入基于 embedding 的 RAG 基础设施。

**log.md** 面向时间。它是一个按时间追加的记录，写下发生了什么、发生在什么时候，包括 ingest、query、lint 等操作。一个实用技巧是：如果每条记录都以一致前缀开头（例如 `## [2026-04-02] ingest | Article Title`），那它就可以用简单的 unix 工具解析，例如 `grep "^## \\[" log.md | tail -5` 可以直接拿到最近 5 条记录。日志让你能看到 wiki 是如何一步步演化出来的，也能帮助 LLM 理解最近完成了哪些工作。

## 可选：CLI 工具

随着 wiki 增长，你可能会想做一些小工具，帮助 LLM 更高效地操作它。最显然的一个需求就是对 wiki 页面做搜索。小规模时，索引文件已经够用了；但随着规模扩大，你可能会想要真正的搜索能力。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个面向 Markdown 文件的本地搜索引擎，支持混合 BM25 / 向量检索，以及 LLM reranking，全都在本地完成。它既有 CLI（LLM 可以直接从 shell 调用），也有 MCP server（LLM 可以原生作为工具使用）。当然，你也可以自己做一个更简单的版本，LLM 完全可以帮你快速 vibe-code 一个朴素搜索脚本，等真的需要时再做。

## Tips and tricks

- **Obsidian Web Clipper** 是一个浏览器扩展，可以把网页文章转换成 Markdown，非常适合快速把资料放进原始来源层。
- **把图片下载到本地。** 在 Obsidian 的 Settings → Files and links 中，把 “Attachment folder path” 设成一个固定目录（例如 `raw/assets/`）。然后在 Settings → Hotkeys 里搜索 “Download”，找到 “Download attachments for current file”，给它绑定一个快捷键（例如 `Ctrl+Shift+D`）。这样在剪藏完一篇文章后，按一下快捷键，就可以把所有图片都下载到本地。这是可选项，但非常有用，因为这样 LLM 就可以直接读取和引用本地图片，而不是依赖可能会失效的外部 URL。需要注意的是，LLM 目前还不能天然地一次性读取“带内嵌图片的 Markdown”。可行的变通方式是：先让 LLM 读取文本，再单独查看部分或全部图片来补充上下文。虽然稍显笨拙，但已经足够实用。
- **Obsidian 的 graph view** 是观察整个 wiki 形态最好的方式：哪些页面相互连接，哪些页面是枢纽，哪些页面成了孤岛。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有插件支持它。你可以直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以基于页面 frontmatter 运行查询。如果你的 LLM 会在页面中写入 YAML frontmatter（例如标签、日期、来源数），Dataview 就可以自动生成动态表格和列表。
- wiki 本质上只是一个由 Markdown 文件组成的 git 仓库。你天然拥有版本历史、分支和协作能力。

## 为什么这套方法有效

维护知识库最烦人的部分，并不是阅读，也不是思考，而是“簿记”工作：更新交叉引用、保持摘要为最新、标注新数据与旧说法之间的矛盾、维护几十个页面之间的一致性。人类之所以会放弃维护 wiki，是因为维护成本增长得比价值更快。LLM 不会觉得无聊，不会忘记更新一个反向链接，而且一次就可以改动 15 个文件。wiki 之所以能保持持续维护，是因为维护成本几乎降到了零。

人类的工作是筛选来源、引导分析、提出好问题，并思考这些材料真正意味着什么。LLM 的工作是除此之外的所有事情。

这个想法在精神上也与 Vannevar Bush 于 1945 年提出的 Memex 有关。Memex 设想的是一种个人的、经过策展的知识存储，并允许文档之间通过联想路径连接起来。从这个角度看，Bush 的愿景其实比今天的 web 更接近这里描述的东西：私人的、主动维护的，而且文档之间的连接和文档本身同样重要。他当时无法解决的问题是谁来维护这些连接。LLM 解决了这个问题。

## 说明

这份文档有意保持抽象。它描述的是一种方法，而不是一个具体实现。具体目录结构、schema 约定、页面格式、工具选择，都要取决于你的领域、你的偏好，以及你使用的 LLM。上面提到的一切都是可选且模块化的：有用的就拿来，不需要的就忽略。例如，如果你的来源都是纯文本，那可能根本不需要图片处理。你的 wiki 如果足够小，可能 `index.md` 就已经够用，不需要搜索引擎。你可能不关心幻灯片，只想要 Markdown 页面。你也可能想输出完全不同的格式。正确的使用方式是：把这份文档分享给你的 LLM agent，然后和它一起把这套模式实例化成适合你自己的版本。这份文档唯一的任务，就是把这种模式讲清楚。剩下的细节，你的 LLM 会和你一起完成。
