# 目标检测

## 页面状态

- 状态：正式 topic
- 事实基座：`wiki/summaries/` 优先

## 主题定义

本页讨论目标检测这一经典视觉任务：在同一张图像中同时回答**“有哪些对象”**与**“这些对象在哪里”**。它的标准输出仍然是类别标签与边界框，因此既不同于只做类别判断的图像分类，也不同于以像素级掩码为目标的实例分割。

这个定义看似简单，但 topic 级别真正需要澄清的是边界。第一，本页讨论的是**不依赖自然语言查询的经典检测主线**，因此不把 grounding、开放词表检测、多模态查询检测直接并入主线。第二，OCR、文档版面分析、表格检测虽然大量借用检测器，但其对象定义、评价协议和下游目标都不完全等同于通用目标检测，因此只能视为邻接问题。第三，本页不把实例分割写成内部子章节，因为当前知识库的证据链还主要集中在边界框检测范式本身。

从现有 summary 组合看，本页最稳的主题定义不是“框出东西的方法大全”，而是：**目标检测如何在 proposal-based pipeline、one-stage 实时接口、set prediction 端到端接口之间重新分配复杂度。** 这也是 `Faster R-CNN`、`YOLO` 系列、`DETR`、`RT-DETR` 与 `Co-DETR` 能够被放进同一页的真正理由。

## 核心问题

- **检测系统是否需要 proposal、anchor、`NMS` 等显式中间结构**，还是可以直接预测最终目标集合。
- **正负样本分配应该是一对多还是一对一**，以及这种训练接口会怎样影响监督密度、优化稳定性与推理简洁性。
- **实时检测的真正瓶颈在哪里**：是 backbone / neck / head 的计算量，还是 `NMS` 等后处理本身的延迟与超参数敏感性。
- **检测性能的关键来源是什么**：更强预训练、更强特征骨干，还是检测任务自身的结构设计与训练分配策略。

## 主线脉络 / 方法分层

从当前 evidence base 看，检测主线不应按“two-stage / one-stage / transformer”做静态分类，而应按**模型把复杂度放在哪一层**来分层。

- **proposal 内化层**：`Faster R-CNN` 的历史地位来自 `RPN`。它没有取消候选框逻辑，而是把候选框生成从外部启发式步骤内化为可学习模块，并与检测头共享卷积特征。这一层的关键是：检测系统仍接受“先提出候选，再做分类回归”的总体框架，但 proposal 不再是外部流水线，而是网络的一部分。也因此，`Faster R-CNN` 代表的不是“老方法”，而是**proposal-based 检测被工程化为统一网络**这一里程碑。
- **one-stage 密集预测层**：`YOLOv1` 把目标检测改写为单阶段回归问题，是第一条真正意义上的实时检测主线起点。`YOLOv2 / YOLO9000` 通过 anchor、dimension clustering、多尺度训练与联合训练，把这一原型推进成更成熟的系列；`YOLOv3` 用多尺度预测和更稳定 backbone 把它推向成熟工程结构；`YOLOv4` 则进一步说明，one-stage 路线在很长一段时间内不是靠范式大跳跃，而是靠系统化工程 recipe 迭代。这一整条线的核心不是“没有中间结构”，而是**把检测复杂度集中在密集预测头和训练 recipe 上，以换取实时推理接口**。
- **set prediction 端到端层**：`DETR` 的真正贡献，是把检测重写成**一对一的集合预测问题**。它不再把检测理解为密集候选框筛选，而是通过 bipartite matching 和 object queries 直接输出最终目标集合。这不是普通组件替换，而是问题建模方式的改变。也因此，`DETR` 对检测主线的冲击主要来自接口层：anchor、手工匹配与 `NMS` 的必要性都被重新质疑。
- **实时 end-to-end 修正层**：`RT-DETR` 说明 `DETR` 并不天然适合实时场景，若要进入 real-time，就必须处理多尺度特征交互、query 初始化和 encoder 计算瓶颈。它的重要性在于证明：**end-to-end detector 不是只能做高精度慢模型**，也可以进入实时检测竞争。与此同时，`YOLOv10` 表明另一条路线也在接近类似目标，即在 `YOLO` 家族内部吸收 end-to-end 思想，通过 dual assignment 推进 `NMS-free` 推理接口。二者共同说明，2024 年前后检测主线的真正变化不是某个家族单独胜出，而是不同家族都开始向“更少后处理、更统一推理接口”靠拢。
- **监督稀疏性回补层**：`Co-DETR` 则提醒我们，一对一 set prediction 的简洁性是有代价的。它指出纯 `DETR` 式一对一匹配会带来监督稀疏，从而削弱 encoder 学习，因此重新引入一对多 assignment 作为训练辅助。这个节点非常重要，因为它表明检测主线并不是单向朝“越纯越端到端”演化，而是在**推理简洁性与训练信号密度之间反复重新平衡**。

若把这几层连起来，当前可以得到一个更有解释力的结论：**目标检测的长期演化，不是从旧范式线性走向新范式，而是不断决定“复杂度应该放在 proposal、dense head、matching 还是后处理上”。** `Faster R-CNN` 把 proposal 内化，`YOLO` 把复杂度压进 one-stage 实时接口，`DETR` 把复杂度移到 set prediction 与 matching，`RT-DETR / YOLOv10 / Co-DETR` 则继续在训练密度、推理简洁性和实时约束之间调参。

## 关键争论与分歧

- **`DETR` 是否已经取代两阶段检测与 `YOLO` 主线**：现有证据不支持这种写法。`DETR` 改写了检测接口，但 `RT-DETR` 仍需做大量实时化修正，`Co-DETR` 也显示一对一监督并非无代价；另一方面，`YOLOv10` 说明 one-stage 家族也在吸收 end-to-end 思想。因此当前更准确的判断是：**接口正在收敛，但范式尚未统一。**
- **`YOLO` 是否只是“快但不够规范”的工程路线**：当前 evidence base 不支持这种刻板结论。`YOLO` 从 `v1` 到 `v10` 的主线恰恰说明，实时检测不是次优分支，而是一个长期独立的问题域。它的问题意识始终是速度、精度、部署复杂度三者的联合优化，而不是单纯牺牲精度换速度。
- **检测应否彻底摆脱 `NMS`**：`DETR`、`RT-DETR`、`YOLOv10` 都把减少或摆脱 `NMS` 当成关键方向，但现有 summary 仍不足以证明所有场景都应该完全抛弃后处理。更稳妥的说法是：**`NMS-free` 已经成为重要目标接口，但尚不能写成无条件共识。**
- **一对一分配是否天然优于一对多分配**：`DETR` 支撑一对一接口的优雅性，`Co-DETR` 则清楚表明一对一也会带来监督稀疏问题。因此这场争论的成立条件非常明确：若你重视推理接口纯度，一对一更吸引人；若你重视训练阶段的密集学习信号，一对多或混合分配仍然有价值。
- **更强预训练是否是检测性能的必要前提**：`Rethinking ImageNet pre-training` 对这一常识构成了重要修正。它至少支持这样一个谨慎结论：检测性能提升并不总是必须依赖大规模分类预训练，尤其当训练时间和目标任务监督足够时。这意味着检测研究不能把所有收益都归因于 backbone 预训练。

## 未解决问题

- 当前页面已经能较稳定地重建 **proposal-based / one-stage / set prediction / real-time end-to-end / 混合 assignment** 五层结构，但 `Deformable DETR`、`DINO`、`RetinaNet`、`FCOS` 等关键节点仍缺 summary，因此检测谱系还未完全补齐。
- `YOLOv5-v9` 在当前页主要由 survey 与官方文档间接承接，缺少逐版原始论文支撑；因此可以写系列连续性，但不宜把每一代的细分贡献写成过强稳定结论。
- 小目标、长尾类别、开放词表检测、视频检测与 grounding detection 目前尚未形成足够 summary 支撑，当前页只能把它们标为邻接方向，不能并入正式主线。
- 评价协议仍主要停留在 `COCO AP` 语境中，部署延迟、稳定性、后处理敏感性等系统层指标虽然在 `RT-DETR` 中开始浮现，但尚未形成独立 comparison 页来稳固这些结论。
