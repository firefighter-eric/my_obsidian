# 目标检测

## 页面状态

- 状态：正式 topic
- 事实基座：`wiki/summaries/` 优先

## 主题定义

本页讨论目标检测这一经典视觉任务：在图像中同时回答“有什么对象”与“对象在哪里”。它的标准输出通常是类别标签与边界框，因此既不同于只做分类的图像识别，也不同于需要像素级掩码的实例分割。

本主题与相邻页面的边界需要明确区分：

- **与 `传统 CV` 的关系**：`传统 CV` 是更上层的总览页；本页只聚焦检测这一条独立主线。
- **与 OCR / 文档版面分析的关系**：文档任务经常借用检测器定位文本框、表格和版面区域，但其对象定义、评价指标与下游目标并不完全等同于通用目标检测。
- **与 grounding / 多模态感知的关系**：grounding 会把语言条件引入检测；本页当前只讨论不依赖自然语言查询的经典检测主线。
- **与实例分割的关系**：实例分割常由检测框架扩展而来，但本页默认只把它视为邻近任务，不展开为核心主线。

## 核心问题

- 检测系统应否依赖显式 proposal、anchor 与手工后处理，还是可以直接端到端预测目标集合。
- 检测训练中的正负样本分配应采用一对多匹配还是一对一匹配；这会如何影响训练稳定性与表示学习。
- 实时检测的主要瓶颈到底来自 backbone、decoder，还是来自 `NMS` 等后处理步骤。
- 检测性能的提升主要来自更强预训练 backbone，还是来自检测任务本身的监督与结构设计。
- 检测是否已经从“复杂工程 pipeline”转向“统一预测接口”，还是仍需要在不同场景下保留多套范式。

## 主线脉络 / 方法分层

从当前证据看，目标检测主线至少可分成五层。

- **proposal-based 两阶段检测**：`Faster R-CNN` 把外部候选框方法替换为 `RPN`，形成共享特征的统一两阶段流程。这条路线的核心不是消灭中间表示，而是把 proposal 变成可学习模块。
- **one-stage 实时检测主线**：`YOLO` 家族从 `YOLOv1` 把检测改写为单阶段回归开始，经 `YOLOv2/9000`、`YOLOv3`、`YOLOv4` 持续在 anchor、特征融合、训练 recipe 与工程效率上演化，确立了实时 one-stage detector 的主流路线。
- **端到端 set prediction**：`DETR` 把检测改写为一对一集合预测，显式削弱 anchor 设计与 `NMS` 的必要性。它改变的不是单个模块，而是检测问题的建模方式。
- **面向实时性的 end-to-end 修正**：`RT-DETR` 与 `YOLOv10` 都说明 end-to-end 实时检测不是单一路径。前者沿 `DETR` 路线逼近实时，后者则在 `YOLO` 家族内部推进 `NMS-free`、一对一推理接口。
- **对一对一监督稀疏性的再平衡**：`Co-DETR` 说明纯 `DETR` 式一对一匹配虽然简化了推理接口，但也会带来训练监督稀疏问题，因此后续路线又重新引入一对多 assignment 作为训练辅助。

这五层之间并不是简单替代关系。更准确的说法是：检测主线从 “外部 proposal + 复杂 pipeline” 逐步分化出 `YOLO` 式 one-stage 实时路线与 `DETR` 式 set prediction 路线，而在 2024 年前后两条路线又开始在 `NMS-free`、一对一分配与部署接口上重新靠近。这个结论由 `Faster R-CNN`、`YOLO` 系列、`DETR`、`RT-DETR`、`YOLOv10` 与 `Co-DETR` 共同支持。

## YOLO 系列的内部脉络

`YOLO` 在当前知识库里已经可以整理成一条较明确的内部演化线：

- **`YOLOv1`**：把检测重写为单阶段回归，确立 “you only look once” 的问题设定。
- **`YOLOv2 / YOLO9000`**：引入 anchor、dimension clustering、多尺度训练与联合 classification/detection，提升 recall、定位与类别扩展能力。
- **`YOLOv3`**：通过 `Darknet-53` 与三尺度预测，把 YOLO 推进为成熟的多尺度 one-stage detector。
- **`YOLOv4`**：以 `CSP`、`Mosaic`、`CIoU`、`PAN` 等组合优化为代表，进入工程 recipe 驱动阶段。
- **`YOLOv5-v9`**：根据 `2025` 综述，这一阶段主要围绕 backbone、neck、head、loss/assignment 与 training strategy 的模块化优化持续演化。
- **`YOLOv10`**：开始把 `YOLO` 家族推进到 `NMS-free`、更接近 end-to-end 的推理接口。
- **`YOLOv11 / YOLO26`**：根据截至 `2026-04-13` 的官方文档，`YOLO11` 仍是稳定生产推荐版本之一，而 `YOLO26` 被官方定义为最新版本。

这一系列说明：`YOLO` 并不是单一论文节点，而是 one-stage 实时检测的长期工程主线。这个判断主要由 `YOLOv1`、`YOLO9000`、`YOLOv3`、`YOLOv4`、`YOLOv10`、`2025` survey 与 `2026` 官方文档共同支持。

## 方法分歧的结构化理解

当前 evidence base 允许把目标检测中的主要分歧整理成三个轴。

- **推理接口轴**：`Faster R-CNN` 代表 proposal-based 接口；`YOLOv1-v9` 代表 one-stage dense prediction 接口；`DETR` 代表 set prediction 接口；`YOLOv10` 则显示 one-stage 家族也开始向一对一、`NMS-free` 接口靠拢。
- **样本分配轴**：经典 detector 与多数 `YOLO` 版本长期依赖一对多匹配；`DETR` 依赖一对一 bipartite matching；`Co-DETR` 与 `YOLOv10` 则说明混合 assignment 或 dual assignment 可能更有利于兼顾训练信号与推理简洁性。
- **速度瓶颈轴**：经典实时 detector 主要优化 backbone、neck 与 head；`RT-DETR` 和 `YOLOv10` 都进一步指出 `NMS` 本身也是实时检测中的显著瓶颈之一。

由此可以得到一个当前较稳的编辑性判断：目标检测的长期演化并不是“谁彻底取代谁”，而是检测系统在统一接口、训练效率、实时约束三者之间不断重新分配复杂度。

## 关键争论与分歧

- **`DETR` 是否已经彻底取代两阶段检测**：现有来源不支持这种结论。`DETR` 改写了接口，但 `Co-DETR` 也显示经典一对多监督思想仍然重要。
- **`YOLO` 是否只是“快但不够准”的工程路线**：现有 `YOLOv2-v4-v10` 证据不支持这种简单判断。`YOLO` 的核心演化一直在围绕速度/精度/部署接口三者同时优化。
- **检测应否完全摆脱 `NMS`**：`RT-DETR` 与 `YOLOv10` 都把摆脱 `NMS` 视为重要方向，但当前知识库尚不足以证明所有场景都该彻底抛弃后处理。
- **实时检测是否等同于 `YOLO` 路线**：从 `RT-DETR` 的论证看，实时性是约束条件而不是某一模型家族的专属属性。
- **更强预训练是否是检测性能的必要前提**：`Rethinking ImageNet pre-training` 对这一常识提出挑战，说明检测性能提升并不总是必须依赖分类预训练。
- **检测是否已经成为“通用视觉基础模型”的附属任务**：当前证据不足。本页仍把它视为具有独立结构与评价逻辑的核心视觉任务。

## 未解决问题

- 当前知识库虽已补到 `YOLOv1 / v2 / v3 / v4 / v10 / v11+官方现状`，但 `YOLOv5-v9` 仍主要依赖综述与官方脉络，没有逐版原始论文支撑。
- `RetinaNet`、`Deformable DETR`、`DINO` 等关键节点仍缺独立 `wiki/summaries/`，因此 one-stage 与 deformable 分支还没有被完整展开。
- 小目标、长尾类别、开放词表检测与视频检测尚未形成独立证据链，暂不宜写成稳定结论。
- 现有材料对评价协议主要停留在 `COCO AP` 语境，尚缺少不同部署目标下的指标比较页。
- 检测与文档版面分析、表格检测、视觉 grounding 的边界仍需后续 comparison 页细化。
