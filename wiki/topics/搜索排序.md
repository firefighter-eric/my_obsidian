# 搜索排序

## 页面状态

- 状态：正式 topic
- 事实基座：`raw/summary/` 优先

## 主题定义

本页讨论搜索系统中的 ranking 问题：给定 query 与一组候选文本，系统如何按“与用户信息需求的相关性”输出有序结果。这里的重点不是通用问答生成，也不是知识库构建本身，而是 query、document、候选集与排序目标之间的建模。它与 [AI 智能问答与智能客服](./AI%20智能问答与智能客服.md) 的边界在于：后者讨论完整问答 / 客服系统，排序只是其中一层；本页则专门收紧到 retrieval stack 内部的排序设计。

搜索排序也不等于单一算法。它通常处在多阶段系统里：先用稀疏或稠密方法做高召回候选生成，再用更强但更慢的模型做重排，必要时继续做 listwise reranking、query/document expansion 或 late interaction 检索。因此，本 topic 的核心不是“哪一个模型最好”，而是搜索排序如何在相关性、延迟、索引成本与部署复杂度之间组织方法层次。

## 核心问题

- 搜索排序中的相关性应如何建模，尤其是在 query 与 document 长度、结构和语言形态明显异质的情况下。
- 搜索系统为什么普遍采用“召回 -> 重排”的多阶段架构，而不是直接让最强模型全库打分。
- transformer 进入排序之后，cross-encoder reranking、bi-encoder dense retrieval 与 late interaction 分别解决了什么问题。
- 长文档、海量候选与严格时延要求下，排序模型如何在效果和效率之间做工程折中。
- 搜索排序应如何与相邻主题切分，避免把检索、问答、RAG 与客服混写成一个泛化大主题。

## 主线脉络 / 方法分层

- 传统 IR 与 neural ranking 背景层：`Mitra, Craswell 2019` 说明搜索排序并不是 transformer 才开始的议题。排序长期建立在向量空间、概率检索与 learning to rank 之上，neural ranking 进一步把相关性建模拆成 query-document 异质性处理、表示建模与交互建模等多个设计轴。编者归纳上，这一层提供的是“排序问题长什么样”的结构框架。
- 前 transformer 的 neural ranking 主线：同一篇综述回顾 `DSSM`、`DRMM`、`K-NRM`、`PACRR`、`DeepRank` 等路线，说明神经排序在 BERT 之前已形成两类重要思路：一类更强调单侧表示学习，另一类更强调局部交互与 query term 级匹配。这意味着搜索排序从一开始就不是纯语义向量问题，而是匹配粒度选择问题。
- BERT cross-encoder 重排主线：`Nogueira, Cho 2019` 证明，只要把 query 与 candidate passage 放进同一个 BERT 输入中做相关性分类，就能显著提升 MS MARCO 等基准效果。这条路线把“重排序”推成 transformer 时代的核心接口：候选先由 BM25 之类高效方法产生，再由高成本交叉编码器精排。
- transformer 排序系统化主线：`Nogueira et al. 2020` 把 transformer 排序归纳为两大体系。一类是多阶段 reranking，包括 `monoBERT`、passage aggregation、cascade 以及 listwise reranking；另一类是直接学习 query/document 表示的 dense retrieval。编者归纳上，这一层把排序研究从单模型比较，推进到完整 search stack 设计。
- 稠密检索直排主线：`Karpukhin et al. 2020` 的 `DPR` 说明，排序并不总是在候选生成之后发生；双塔表征本身就可以承担召回与初排功能。对当前 topic 而言，`DPR` 支撑的是“排序边界向召回前移”这一结构变化，而不是替代重排序。
- late interaction 折中主线：`Khattab, Zaharia 2020` 的 `ColBERT` 提出 query/document 分开编码、token 级 `MaxSim` 交互的 late interaction 路线。它既不同于每个候选都完整联合编码的 cross-encoder，也不同于只比较单向量的 bi-encoder，而是在效果与效率之间提供中间解。
- 排序与问答 / 客服系统的接口层：`Sakata et al. 2019` 和 `DPR` 共同说明，在 FAQ 检索、开放域 QA 与知识库问答里，排序经常不是终点，而是后续答案抽取或生成的入口。这里需要明确区分：本页关注“如何把候选排对”，不直接讨论最终回答策略。

## 关键争论与分歧

- 排序的核心应是表示学习还是交互建模：`Mitra, Craswell 2019` 表明这不是简单的二选一。纯表示法更利于索引和扩展，交互法通常相关性更强，但成本也更高。
- cross-encoder 是否应成为默认最优解：`Nogueira, Cho 2019` 强力支持其效果优势，但 `ColBERT` 与 dense retrieval 路线说明，一旦进入真实搜索系统，延迟和索引约束会迫使方法分层。
- dense retrieval 是不是会取代 reranking：`DPR` 证明稠密表征能显著改造召回层，但 `BERT and Beyond` 与 `ColBERT` 都说明高效果搜索仍常依赖后续重排或更细粒度交互，因此更合理的理解是“分工变化”，不是单向替代。
- 排序是否应直接等同于搜索：不能。搜索是更大的系统问题，还包括索引、候选生成、去重、多样性、展示和后续生成模块；本页只覆盖 relevance ranking 主线。

## 未解决问题

- 当前知识库还没有补入更直接的经典 `LTR` 来源，例如 `RankNet / LambdaMART`，因此“搜索排序如何从 feature-based LTR 演进到 neural ranking”仍有一段历史链路缺口。
- 现有证据更偏 passage ranking、QA retrieval 与 text ranking，对 web search 中的多样性排序、商业目标混排与点击偏差校正支撑仍不足。
- `ColBERT` 之后的多向量检索、listwise LLM reranking 与生成式搜索排序，目前仓库里还缺少稳定 summary，暂不宜写成正式结论。
