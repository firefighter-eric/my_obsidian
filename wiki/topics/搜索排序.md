# 搜索排序

## 页面状态

- 状态：正式 topic
- 事实基座：`wiki/summaries/` 优先

## 主题定义

本页讨论搜索系统中的 **relevance ranking** 问题：给定 query 与候选文档集合，系统如何依据用户信息需求输出有序结果。这里的核心不是一般性的“检索都算搜索排序”，而是 **在候选已存在或可被生成的前提下，如何定义、学习并部署相关性排序函数**。这使本页与知识库构建、索引工程、问答生成和客服执行等相邻问题保持必要边界。

搜索排序也不应被压缩成某一种模型名称。它通常存在于多阶段系统中：先有高召回的候选生成，再有成本更高但判断更精细的重排，必要时还会加入 late interaction、listwise reranking 或 task-specific aggregation。因此，本 topic 关注的不是“哪一种模型赢了”，而是 **排序系统如何围绕相关性、延迟、索引成本和交互粒度形成方法分层**。

它与 [AI 智能问答与智能客服](./AI%20智能问答与智能客服.md) 的边界在于：客服把排序视为系统中的一层，而本页专门讨论排序本身。它与 [传统 NLP](./传统%20NLP.md) 的边界在于：传统 NLP 会把检索视作更宽的表示学习分支，而本页收紧到 **query-document relevance ranking** 的方法组织。它与 RAG 的边界在于：RAG 关心召回内容如何被后续生成模块利用，而本页只讨论候选如何被排对。

## 核心问题

- **相关性到底如何建模**：搜索中的 query 与 document 长度、语义粒度和结构都高度异质，因此排序问题并不只是分类问题的平移。
- **为什么搜索系统普遍采用多阶段架构**：如果最强模型不能直接全库运行，那么召回层与重排层的功能边界必须被明确定义。
- **cross-encoder、bi-encoder 与 late interaction 分别解决了什么问题**：它们不是简单的快慢替代，而是对“何时发生交互、交互粒度多细”给出不同答案。
- **排序为何不能等同于检索整体**：排序只是搜索系统的一部分，还必须与索引、召回、多样性、展示和后续生成接口区分开。
- **效果与效率如何折中**：现实系统中最优排序方案往往不是单点效果最高，而是整体吞吐、延迟和成本约束下的最优分层。

## 主线脉络 / 方法分层

- **问题结构与神经排序背景层**：`Mitra, Craswell 2019` 提供了一个仍然有效的框架，即排序长期面临 query-document 异质性、相关性粒度和匹配信号聚合这几个核心问题。它的重要性在于提醒我们：**搜索排序不是从 BERT 才开始，而是本来就有完整的 neural ranking 问题结构**。Transformer 进入排序，只是把这些旧问题放到新的表征与交互机制上重写。
- **表示学习与交互建模分化层**：同一综述回顾的 `DSSM`、`DRMM`、`K-NRM`、`PACRR`、`DeepRank` 等路线，已经清楚展示出两种主思路：一类优先学习 query 与 document 的单侧表示，一类优先在词项或局部片段层面显式建模交互。这一点很关键，因为后续 transformer 排序中的 bi-encoder 与 cross-encoder 之争，本质上就是 **表示优先与交互优先** 的延续，而不是突然出现的新分歧。
- **cross-encoder 重排层**：`Nogueira, Cho 2019` 的 `Passage Re-ranking with BERT` 使 transformer 排序真正进入主流。它把 query 与 passage 联合输入 BERT，在完整交互条件下输出相关性分数。这一路线的核心价值不只是“效果更好”，而是明确建立了 **高召回候选生成之后再用强交互模型精排** 的现代标准结构。它也同时暴露了明显限制：因为每个候选都要和 query 重新联合编码，所以它天然难以承担全库检索职责。
- **多阶段 transformer 排序系统层**：`Nogueira et al. 2020` 进一步把 transformer 排序整理成一个更完整的 search stack，而不只是一篇单模型论文。它说明，`monoBERT`、passage aggregation、cascade 和 listwise reranking 等路线其实都在回答同一件事：**如何把强相关性判断嵌入可部署的多阶段系统**。这使排序问题从“模型好不好”转向“系统如何分层”。
- **bi-encoder 稠密检索层**：`Karpukhin et al. 2020` 的 DPR 代表另一种选择，即把排序边界前移，让双塔表征本身承担召回甚至初排责任。这里 query 和 document 分别编码，得以建立可索引向量空间。它解决的不是最强精排，而是 **在海量语料下保留语义召回能力**。因此 DPR 对本 topic 的重要性不在于“它是不是最终排序模型”，而在于它改变了排序与召回的边界。
- **late interaction 折中层**：`Khattab, Zaharia 2020` 的 `ColBERT` 提供了一条介于 cross-encoder 与 bi-encoder 之间的路线。它保持 query/document 分开编码的索引友好性，同时在 token 级使用 `MaxSim` 做更细粒度交互。`ColBERT` 的价值在于说明，排序并不是只能在“全交互高效果”和“单向量高效率”之间二选一，而可以通过 **延后交互、保留多向量信息** 获得更细腻的折中。
- **问答与 FAQ 接口层**：`Sakata et al. 2019` 和 `DPR` 共同说明，排序常常不是系统终点，而是开放域问答、FAQ 检索和客服系统的入口层。这里必须强调边界：本页只讨论 **如何把候选排到合理顺序**；至于最终答案如何抽取、生成或执行，则属于问答、客服或 RAG 的上层问题。

## 关键争论与分歧

- **排序的核心应由表示学习主导，还是由交互建模主导**：`Mitra, Craswell 2019` 已经说明这不是抽象哲学之争，而是部署约束与相关性精度之间的结构矛盾。表示优先方法适合索引和扩展，但容易损失细粒度匹配；交互优先方法通常效果更强，但成本更高。只有在系统目标明确时，这个争论才有可判定性。
- **cross-encoder 是否应被视为默认最优排序器**：若问题是小候选集精排，`Nogueira, Cho 2019` 的证据很强；但一旦进入海量候选、低延迟和在线服务场景，`DPR` 与 `ColBERT` 就显示出显著优势。因此更稳妥的结论是：**cross-encoder 更像精排上界，而不是通用系统默认解**。
- **dense retrieval 会不会替代 reranking**：现有证据不支持简单替代论。`DPR` 改变了召回层，`ColBERT` 提供了中间折中，`BERT and Beyond` 则显示高性能系统仍然普遍需要后续重排。更合理的理解是：**dense retrieval 重新分配了排序栈中的职责，但没有消灭重排需求**。
- **排序能否直接等同于搜索整体**：不能。若把排序写成整个搜索系统，就会把索引、多样性、去重、点击建模、展示策略和生成接口都混进来。当前证据面足以支撑 relevance ranking 主线，但不足以支撑“完整搜索系统综述”，因此必须收紧边界。

## 证据基础

- [Mitra, Craswell - 2019 - A Deep Look into Neural Ranking Models for Information Retrieval](../../wiki/summaries/Mitra,%20Craswell%20-%202019%20-%20A%20Deep%20Look%20into%20Neural%20Ranking%20Models%20for%20Information%20Retrieval.md)：支撑搜索排序的问题结构、神经排序主分化线以及表示/交互之争的背景。
- [Nogueira, Cho - 2019 - Passage Re-ranking with BERT](../../wiki/summaries/Nogueira,%20Cho%20-%202019%20-%20Passage%20Re-ranking%20with%20BERT.md)：支撑 BERT cross-encoder 重排成为 transformer 时代核心接口。
- [Karpukhin et al. - 2020 - Dense passage retrieval for open-domain question answering](../../wiki/summaries/Karpukhin%20et%20al.%20-%202020%20-%20Dense%20passage%20retrieval%20for%20open-domain%20question%20answering.md)：支撑双塔表征把排序边界前移到召回层。
- [Khattab, Zaharia - 2020 - ColBERT Efficient and Effective Passage Search via Contextualized Late Interaction over BERT](../../wiki/summaries/Khattab,%20Zaharia%20-%202020%20-%20ColBERT%20Efficient%20and%20Effective%20Passage%20Search%20via%20Contextualized%20Late%20Interaction%20over%20BERT.md)：支撑 late interaction 作为效果与效率折中的独立路线。
- [Nogueira et al. - 2020 - Pretrained Transformers for Text Ranking BERT and Beyond](../../wiki/summaries/Nogueira%20et%20al.%20-%202020%20-%20Pretrained%20Transformers%20for%20Text%20Ranking%20BERT%20and%20Beyond.md)：支撑 transformer 排序从单模型走向多阶段系统化组织。
- [Sakata et al. - 2019 - FAQ retrieval using query-question similarity and BERT-based query-answer relevance](../../wiki/summaries/Sakata%20et%20al.%20-%202019%20-%20FAQ%20retrieval%20using%20query-question%20similarity%20and%20BERT-based%20query-answer%20relevance.md)：支撑 FAQ 与问答场景中排序作为入口层的具体实例。

## 未解决问题

- **经典 learning-to-rank 历史链仍不完整**：当前页面已经能写清 neural ranking 到 transformer 排序的主线，但由于缺少 `RankNet`、`LambdaMART` 等更经典 LTR summary，还不能完整说明搜索排序如何从 feature-based LTR 演化到当前神经方法。
- **真实搜索系统中的多目标排序尚缺支撑**：现有 summary 更偏 passage ranking、open-domain QA 与文本相关性，对 web search 中的多样性、商业目标、点击偏差校正和会话式搜索覆盖不足，因此本页不能把这些内容写成稳定结论。
- **LLM reranker 与生成式搜索排序仍属证据空白**：当前知识库尚未补入足够 summary，去支撑 listwise LLM reranking、生成式搜索代理或 multi-vector retrieval 新阶段的成熟判断，因此这里只能明确标注为待补缺口。
